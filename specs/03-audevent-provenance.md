# Phase 3: AuditEvent Provenance Architecture (Weeks 5–6)

> **Architecture lock (June 2026):** The AuditEvent payload carries **only EXT 6
> (`ingest-pipeline-id`) and EXT 7 (`ol-run-id`)**. Source/feed/type/SSoR
> attribution — what older drafts placed in EXT 1–5 — lives in **Unity Catalog
> table properties** (generated by `dqar-client-kit`, loaded by this kit). There
> is **no source-inference algorithm**. A resource whose feed cannot be resolved
> against the manifest is `UNKNOWN` = Tier 1 finding, surfaced by client-kit's
> manifest matching. This supersedes any seven-extension/inference description in
> earlier revisions of this file.

Write a lean, two-extension AuditEvent for every ingested resource, paired with
the resource in a single atomic transaction bundle.

---

## The Two Extensions

| EXT | URL suffix | Set by | Value | Purpose |
|---|---|---|---|---|
| 6 | `ingest-pipeline-id` | orchestrator | valueString | Identifies the ingest job that wrote the resource |
| 7 | `ol-run-id` | orchestrator | valueString (UUID) | The OpenLineage RunEvent that produced the resource — an ingest batch tag, **not** a join key |

No other extensions are written on the AuditEvent. Do **not** add `source-type`,
`source-system-id`, `source-feed-id`, `source-inference-confidence`, or
`ecds-ssor` here — those are UC table properties (see Phase 1 / client-kit).

---

## Deliverables

1. **AuditEventBuilder** (`dqar_aidbox_databricks_kit/provenance/audevent_builder.py`)
   - Build a two-extension AuditEvent for a given resource + `IngestContext`
   - Validate that only EXT 6 + 7 are present (reject any source-attribution extension)
2. **IngestContext** (`dqar_aidbox_databricks_kit/provenance/ingest_context.py`)
   - Carry `ingest_pipeline_id` (EXT 6) and `ol_run_id` (EXT 7) for the batch
   - `ol_run_id` is set from the OpenLineage RunEvent emitted in Phase 2
3. **TransactionBundleAssembler** (`dqar_aidbox_databricks_kit/provenance/bundle_assembler.py`)
   - Pair each resource with its AuditEvent into one `type: "transaction"` bundle
   - POST to Aidbox; rely on in-bundle atomicity (resource + AuditEvent together)
4. **Tests** (`tests/test_audevent_provenance.py`)
   - Exactly two extensions present, both EXT 6 + 7
   - Any source-attribution extension is rejected
   - `entity.what.reference` resolves to the paired resource via `urn:uuid:`
   - Bundle is `transaction`, never `batch`

```json
{

  "outcome": "0",

  "extension": [
    { "url": "http://sonian.io/fhir/ext/ingest-pipeline-id", "valueString": "interbox-job-20251014-ehr-001" },
    { "url": "http://sonian.io/fhir/ext/ol-run-id", "valueString": "a1b2c3d4-5e6f-7a8b-9c0d-1e2f3a4b5c6d" }
  ]
}
```

## Example Atomic Transaction Bundle

```json
{
  "resourceType": "Bundle",
  "type": "transaction",
  "entry": [
    {
      "fullUrl": "urn:uuid:obs-temp",
      "request": { "method": "POST", "url": "Observation" },
      "resource": {
        "resourceType": "Observation",
        "status": "final",
        "code": { "coding": [{ "system": "http://loinc.org", "code": "4548-4" }] },
        "subject": { "reference": "Patient/p1" },
        "valueQuantity": { "value": 7.2, "unit": "%" }
      }
    },
    {
      "request": { "method": "POST", "url": "AuditEvent" },
      "resource": {
        "resourceType": "AuditEvent",
        "action": "C",
        "recorded": "2025-10-14T21:45:00Z",
        "outcome": "0",
        "entity": [{ "what": { "reference": "urn:uuid:obs-temp" } }],
        "extension": [
          { "url": "http://sonian.io/fhir/ext/ingest-pipeline-id", "valueString": "interbox-job-20251014-ehr-001" },
          { "url": "http://sonian.io/fhir/ext/ol-run-id", "valueString": "a1b2c3d4-5e6f-7a8b-9c0d-1e2f3a4b5c6d" }
        ]
      }
    }
  ]
}
```

Rules:
- One bundle = one resource + one AuditEvent. Provenance is per-resource.
- `type: "transaction"` (atomic), never `batch` (independent entries).
- AuditEvent `entity.what.reference` uses the **same `urn:uuid:`** placeholder as
  the resource `fullUrl` — Aidbox resolves both to the real ID in-transaction.
- Aidbox atomicity applies only within one bundle POST — never split the pair.

## Builder Sketch

```python
class AuditEventBuilder:
    """Builds the two-extension AuditEvent. Rejects source-attribution extensions."""

    EXT_PIPELINE = "http://sonian.io/fhir/ext/ingest-pipeline-id"
    EXT_OL_RUN   = "http://sonian.io/fhir/ext/ol-run-id"

    def build(self, resource_full_url: str, ctx: "IngestContext") -> dict:
        return {
            "resourceType": "AuditEvent",
            "action": "C",
            "recorded": ctx.recorded_iso,
            "outcome": "0",
            "entity": [{"what": {"reference": resource_full_url}}],
            "extension": [
                {"url": self.EXT_PIPELINE, "valueString": ctx.ingest_pipeline_id},
                {"url": self.EXT_OL_RUN,   "valueString": ctx.ol_run_id},
            ],
        }
```

```python
class IngestContext:
    """Per-batch provenance context. ol_run_id == the Phase 2 RunEvent runId."""
    def __init__(self, ingest_pipeline_id: str, ol_run_id: str, recorded_iso: str):
        self.ingest_pipeline_id = ingest_pipeline_id   # EXT 6
        self.ol_run_id = ol_run_id                     # EXT 7 — from OpenLineage start_run()
        self.recorded_iso = recorded_iso
```

## What This Phase Is and Is Not

**Is:** per-resource ingest-run identity (EXT 6) + lineage-run linkage (EXT 7);
HIPAA audit-control baseline (45 CFR §164.312(b)); the anchor for re-ingest-by-job
and resolve-origin-RunEvent-in-OpenMetadata.

**Is not:** source attribution (UC properties); measure-level provenance; coverage
for resources created outside the managed pipeline; a join key (`ol-run-id` tags a
batch — the graph is assembled in OpenMetadata).

**Regulatory status:** no current CMS/NCQA mandate — position as Tier 3 digital
readiness. NCQA removed the ECDS SSoR *submission* requirement as of MY2026, but
internal source attribution (in UC properties) still aids provenance and risk
stratification.

## Phase 3 Success Criteria

- [ ] AuditEvent carries exactly two extensions (EXT 6 + 7), validated
- [ ] Builder rejects any source-type/system/feed/confidence/ssor extension
- [ ] Resource + AuditEvent written in one `transaction` bundle (atomic)
- [ ] `entity.what.reference` resolves to the paired resource
- [ ] `ol_run_id` equals the Phase 2 RunEvent `runId` (matches OpenMetadata)
- [ ] No source-inference code path exists anywhere in the kit
