## What This Component Does

Bridges DQAR conformance results and FHIR provenance into the analytics stack:

1. Writes write-time **AuditEvent provenance** (two extensions) for ingested resources
2. Emits **OpenLineage RunEvents** (with `DQARIngestFacet` field mappings) to OpenMetadata
3. Loads **UC table properties** generated by `dqar-client-kit` into Databricks Unity Catalog

It consumes client-kit's output; it does not run conformance checks or generate UC
property values itself.

## Input → Output

```
dqar-client-kit output (uc-properties.json)
   +  ingested FHIR resources
        ↓
dqar-aidbox-databricks-kit
        ↓
Output:
- Resource + AuditEvent pairs written to Aidbox (atomic bundles)
- OpenLineage RunEvents emitted to OpenMetadata
- UC table properties applied to Databricks
```

## Key Design Decisions

- ✅ **AuditEvent carries EXT 6 + 7 only** (`ingest-pipeline-id`, `ol-run-id`) — source/feed/type/SSoR attribution lives in **UC table properties**, not the AuditEvent
- ✅ **No source inference** — explicit manifest matching only; undeclared = `UNKNOWN` = Tier 1 finding
- ✅ Atomic transaction bundles — every resource + AuditEvent written in one bundle POST
- ✅ OpenLineage as the lineage backbone protocol
- ✅ **OpenMetadata as the single lineage backend** — Marquez dropped; RunEvents emitted directly; no `lineage_resource_map` join table
- ✅ `ol-run-id` is an ingest batch tag and graph **lookup key**, not a relational join key
- ✅ Databricks Unity Catalog as the analytics catalog target

> Supersedes earlier "seven-extension AuditEvent provenance" framing. Source
> attribution moved to UC properties; the AuditEvent payload is two extensions.

## The Three Linkages (the kit's core invariant)

The kit maintains three consistent linkages; drift between any pair is a finding:

- AuditEvent **EXT 7 `ol-run-id`** == OpenLineage RunEvent **`runId`**
- `DQARIngestFacet.sourceFeedId` == UC **`dqar_source_feed_id`**
- UC properties (per-table attribution) ←→ AuditEvent (per-resource run identity)
