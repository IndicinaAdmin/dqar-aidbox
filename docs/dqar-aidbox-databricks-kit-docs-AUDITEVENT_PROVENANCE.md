## The Lean AuditEvent Contract

This kit writes a **two-extension** AuditEvent at ingest time. The AuditEvent payload carries only the two run-time tags the pipeline orchestrator owns:

| EXT | URL suffix | Set by | Value type | Purpose |
|---|---|---|---|---|
| 6 | `ingest-pipeline-id` | orchestrator | valueString | Identifies the ingest job that wrote the resource |
| 7 | `ol-run-id` | orchestrator | valueString (UUID) | The OpenLineage RunEvent that produced this resource — an ingest batch tag, **not** a join key |

That is the entire extension set on the AuditEvent. There is no source-type, source-system-id, source-feed-id, inference-confidence, or ECDS-SSoR extension on the AuditEvent payload.

### Where source attribution lives instead

Source/feed/type/SSoR attribution — what older designs placed in AuditEvent EXT 1–5 — is expressed as **Unity Catalog table properties** in Databricks, generated by `cdar-client-kit` and loaded by this kit. See `UC_PROPERTIES_LOADING.md` and the client-kit `UC_PROPERTIES.md` for the full key set (`dqar_source_system_id`, `dqar_source_feed_id`, `dqar_source_type`, `dqar_ecds_ssor_category`, and the conformance/findings keys).

This split is deliberate:

- **AuditEvent (EXT 6 + 7)** answers *"which ingest run produced this exact resource, and which OpenLineage RunEvent describes that run?"* — per-resource, write-time, owned by the orchestrator.
- **UC table properties** answer *"what feed/system/type/SSoR does this table's data come from, and how conformant is it?"* — per-table, catalog-side, owned by the assessment.

No source-inference algorithm exists anywhere in the pipeline. A resource whose feed cannot be resolved against the manifest is `UNKNOWN`, and an `UNKNOWN` feed is a Tier 1 governance finding — surfaced through the manifest-matching layer in client-kit, not through an AuditEvent extension.

## Extension Definitions

### EXT 6 — `ingest-pipeline-id`

```json
{
  "url": "http://sonian.io/fhir/ext/ingest-pipeline-id",
  "valueString": "interbox-job-20251014-ehr-001"
}
```

- **Format:** `{engine}-job-{yyyymmdd}-{feed-shortname}-{seq}`
- **Owner:** pipeline orchestrator (Interbox job runner), set at write time
- **Stability:** stable per ingest job; every resource in the same job carries the same value
- **Use:** correlates all resources written by one ingest run; the anchor for "re-ingest the resources from job X"

### EXT 7 — `ol-run-id`

```json
{
  "url": "http://sonian.io/fhir/ext/ol-run-id",
  "valueString": "a1b2c3d4-5e6f-7a8b-9c0d-1e2f3a4b5c6d"
}
```

- **Format:** UUID — the exact `run.runId` of the OpenLineage RunEvent emitted for the ingest batch (see `OPENLINEAGE_EMISSION.md`)
- **Owner:** orchestrator, set at write time from the RunEvent it emits
- **Critical semantics:** this is an **ingest batch tag, not a foreign key**. The lineage graph is assembled in OpenMetadata from each RunEvent's declared inputs and outputs. You do not join resources to lineage on `ol-run-id`; you use it to look up *which* RunEvent's declared I/O describes a resource's origin.
- **Coverage requirement:** valid `ol-run-id` values that resolve to a real RunEvent in the OpenLineage graph are required for CDAR provenance-maturity Level 3+.

## The Atomic Transaction Bundle Pattern

Every clinical resource and its AuditEvent must be written in a **single FHIR transaction bundle POST**. Aidbox's atomicity guarantee applies only within one bundle POST — split the resource and its AuditEvent across two requests and you can end up with a resource that has no provenance record (or vice versa) if one call fails.

```json
{
  "resourceType": "Bundle",
  "type": "transaction",
  "entry": [
    {
      "fullUrl": "urn:uuid:obs-temp",
      "request": { "method": "POST", "url": "Observation" },
      "resource": {
        "resourceType": "Observation",
        "status": "final",
        "code": { "coding": [{ "system": "http://loinc.org", "code": "4548-4" }] },
        "subject": { "reference": "Patient/p1" },
        "valueQuantity": { "value": 7.2, "unit": "%" }
      }
    },
    {
      "request": { "method": "POST", "url": "AuditEvent" },
      "resource": {
        "resourceType": "AuditEvent",
        "type": { "system": "http://terminology.hl7.org/CodeSystem/audit-event-type", "code": "rest" },
        "action": "C",
        "recorded": "2025-10-14T21:45:00Z",
        "outcome": "0",
        "entity": [{ "what": { "reference": "urn:uuid:obs-temp" } }],
        "extension": [
          { "url": "http://sonian.io/fhir/ext/ingest-pipeline-id", "valueString": "interbox-job-20251014-ehr-001" },
          { "url": "http://sonian.io/fhir/ext/ol-run-id", "valueString": "a1b2c3d4-5e6f-7a8b-9c0d-1e2f3a4b5c6d" }
        ]
      }
    }
  ]
}
```

Key points:

- The AuditEvent's `entity.what.reference` uses the **same `urn:uuid:` placeholder** as the resource's `fullUrl`. Aidbox resolves both to the real resource ID inside the transaction, so the AuditEvent points at the resource it describes without pre-generating IDs.
- One bundle = one resource + one AuditEvent. Do not batch many resources with one shared AuditEvent — provenance is per-resource.
- Use `type: "transaction"`, never `type: "batch"`. Batch entries succeed/fail independently; that breaks the resource↔AuditEvent atomicity guarantee.

## What AuditEvent Logging Is and Is Not

**Is:**
- ✅ Closes the HIPAA audit-control gap (45 CFR §164.312(b))
- ✅ Establishes a FHIR-server transaction-log baseline
- ✅ Provides per-resource ingest-run identity (EXT 6) and lineage-run linkage (EXT 7)
- ✅ Lets you re-ingest or invalidate by job, and resolve a resource's origin RunEvent in OpenMetadata

**Is not:**
- ❌ Not a current CMS or NCQA mandate — position as a Digital Readiness (Tier 3) capability, not a Tier 1 governance gap
- ❌ Not source attribution — that lives in UC table properties, not here
- ❌ Not measure-level execution provenance — it records ingest, not measure calculation
- ❌ Not coverage for resources created outside the managed ingest pipeline
- ❌ Not a join key — `ol-run-id` tags a batch; the graph is assembled in OpenMetadata

---

## Regulatory Status

No specific CMS or NCQA mandate requires AuditEvent logging today. As of MY2026, NCQA removed the SSoR *submission* requirement for ECDS — but a plan still benefits from internal source attribution (held in UC properties) for provenance and risk stratification. Position AuditEvent logging as a Tier 3 digital-readiness capability with MY2029 and the Jan 1, 2027 CMS-0057-F P2P deadline attached.

## Implementation Surface in This Kit

| Component | Responsibility |
|---|---|
| `AuditEventBuilder` | Builds the two-extension AuditEvent for a given resource + ingest context |
| `TransactionBundleAssembler` | Pairs each resource with its AuditEvent into one transaction bundle |
| `IngestContext` | Carries `ingest-pipeline-id` and `ol-run-id` for the current run, set by the orchestrator |

The orchestrator obtains `ol-run-id` from the OpenLineage RunEvent it emits for the batch (see `OPENLINEAGE_EMISSION.md`), then threads it through `IngestContext` so every AuditEvent in the batch carries the matching UUID.

## Best Practices

1. **Set both extensions or neither.** A resource with EXT 6 but no EXT 7 cannot be located in the lineage graph; one with EXT 7 but no EXT 6 cannot be re-ingested by job. Treat them as a pair.
2. **Never invent an `ol-run-id`.** It must equal the `runId` of a RunEvent actually emitted to OpenMetadata. A UUID that resolves to nothing fails provenance-maturity Level 3+.
3. **Keep source attribution out of the AuditEvent.** If you find yourself adding a `source-type` extension, stop — that belongs in UC properties. The AuditEvent stays at two extensions.
4. **One bundle per resource+AuditEvent pair.** Resist the urge to optimize into multi-resource bundles; the atomicity guarantee is the whole point.
